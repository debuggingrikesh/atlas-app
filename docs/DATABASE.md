# Database Conventions & Documentation

## Overview

Project Atlas uses Supabase-hosted PostgreSQL with Prisma ORM.

The schema is defined in `prisma/schema.prisma`. 
Migrations are handled via `prisma migrate dev`.

## Architectural Constraints

### Supabase Auth ↔ UserProfile Sync

> [!WARNING]
> Users exist in two places: Supabase Auth (identity) and Prisma `UserProfile` (application data).

When a user signs up via Supabase Auth, a corresponding row **must** be created in the `UserProfile` table.
The `UserProfile.id` must match the Supabase Auth `user.id` exactly (it is not auto-generated by Prisma).

### Multi-tenant Data Isolation

> [!IMPORTANT]
> All queries to tenant-specific tables must include `businessId` in the `where` clause.

- Do not expose raw Prisma clients to page components or API routes directly.
- Use data-access functions in module `lib/` directories that enforce the `businessId` parameter.

## Tables

### IndustryTemplate
Pre-defined vertical templates (e.g., Healthcare Clinic, Education).
- `slug` is used for programmatic lookups.
- Seeded via `prisma/seed.ts`.

### Business
The core tenant entity.
- Must belong to an `IndustryTemplate`.

### Branch
Physical or logical locations for a Business.

### UserProfile
Application-level user profile (name, avatar).
- `id` maps to Supabase Auth `id`.

### BusinessMember
Join table connecting `UserProfile` to `Business`.
- Defines the user's `MemberRole` (OWNER, ADMIN, MEMBER).

### AuditLog
Immutable, append-only log of system actions.
- Used for tracking mutations (e.g. `business.created`).
- Cannot be updated or deleted.
